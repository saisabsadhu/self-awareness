# Audit Report: RB + SA Answer-or-Abstain System

Author: Saisab Sadhu

This report covers the `sa-module-v2` codebase that was shared along with the project brief, and is cross-checked against all the result files present in the Drive folder. Of the four benchmarks discussed in the README, only two, namely SelfAware and FalseQA, ship with code that can actually be run, and the analysis below is therefore grounded most firmly in those two. Wherever the other benchmarks are discussed, it is on the basis of the result files alone, since no corresponding source code was found anywhere in the shared materials.

## 1. What the code does

The system builds a Self-Awareness layer, referred to throughout as SA, on top of an existing memory framework called ReasoningBank, referred to as RB. The overall idea is fairly simple to state even though the implementation has a fair number of moving parts. For a given benchmark such as SelfAware, the model in use, Qwen2.5-7B-Instruct, is first asked each training question once, at temperature zero, with a system prompt that already nudges it to hedge on subjective or unanswerable questions. Whatever the model says in response is then run through a deterministic labelling function that checks the text for roughly forty hedge phrases such as "I don't know", "it depends" and "no consensus", and compares this against the dataset's own ground truth label for whether the question was answerable in the first place. Out of this comparison, every attempt is assigned exactly one of four outcomes: CORRECT, KNOWLEDGE_BOUNDARY_MISSED for a question that was unanswerable but was attempted anyway, FALSE_IDK for a question that was answerable but was wrongly declined, and FACTUAL_ERROR for a question that was attempted but answered incorrectly. It is worth noting that this labelling is entirely rule based and is never asked of the model itself.

From every such episode, two kinds of memory are extracted. The first is a generic RB "strategy" note, following the original ReasoningBank convention of writing separate prompts for successes and failures. The second is an SA "lesson", extracted through a prompt that is specific to the failure type in question, so that a FALSE_IDK lesson reads roughly as "here is why you should have attempted this", while a KNOWLEDGE_BOUNDARY_MISSED lesson reads as "here is why this question could not be answered". In both cases, the failure type itself is handed to the extraction prompt rather than left for the model to infer. These episodes accumulate in a low-level episodic store, and every twenty episodes the system consolidates them, grouping by the pair of failure type and a free-text one-word domain tag that the model generated during extraction, and promoting any group that has recurred at least three times into what the codebase calls a standing guideline. These guidelines sit at a higher level than individual episodes and are meant to be injected into every subsequent prompt, not merely retrieved when a similar question comes along.

At test time, the same held-out questions are run under three conditions that build on each other. The baseline condition is simply the bare question with no memory of any kind. The second condition, called with_memory, adds the top three ReasoningBank strategy notes retrieved by cosine similarity, using the Qwen3-Embedding-0.6B model for the embeddings. The third condition, with_memory_sa, adds the top two ReasoningBank notes together with the top two retrieved SA failure episodes, successes having been deliberately filtered out of SA retrieval, as well as all currently active standing guidelines, all of this being injected into the system prompt at once. Each response under each condition is scored using the same deterministic labelling function described earlier, and an F1 score is computed by treating "the model should have abstained" as the positive class, so that precision is hurt by over-abstention and recall is hurt by under-abstention. The FalseQA script follows this same structure almost line for line, the only real difference being that it is built around false premises rather than genuinely unanswerable questions.

## 2. How the pipeline runs from dataset to metric

The dataset arrives as a pre-split pair of JSON files for training and testing, and there is no code anywhere in the shared materials to regenerate this split from a raw source, so it has to be taken as given. During training, the loop is ask, label deterministically, extract an RB note and an SA lesson, and store both; this training pass is the only point in the entire pipeline where the ground truth label actually touches the system. Every twenty episodes, the system consolidates and checks whether any failure pattern has recurred often enough to be promoted into a standing guideline. The test pass then repeats across the three conditions on the same fixed held-out set, always at temperature zero, and for each item the model is asked with whatever context that condition calls for, after which the same detector labels the outcome again. From there, accuracy, per-class accuracy, precision, recall, F1 and a Wilson confidence interval are all aggregated and written out to the results folder, from where the numbers eventually make their way into the README.

Two things about this flow deserve to be said plainly rather than left implicit. The first is that the very same keyword-based labelling function is used both to generate the training signal and to score the test-time outcome, so there is no independent judge anywhere in the loop checking whether that function is actually correct. The second is that the F1 score measures only the decision of whether to answer or abstain, and says nothing about whether an attempted answer was actually right, since a FACTUAL_ERROR still counts as the correct decision for the purposes of this metric, given that attempting was the right call to begin with. This is a defensible scope choice given how narrowly the brief frames the problem, but it does mean that the headline number, taken on its own, says nothing whatsoever about the quality of the answers being given.

## 3. Assumptions the system makes

Several assumptions are baked into this design, and it is worth laying them out explicitly rather than letting them sit unexamined. The first is that a list of about forty keyword phrases can reliably tell the difference between a model that has genuinely expressed uncertainty and one that has not. The second, following directly from the first, is that this same detector is considered trustworthy enough to serve as both the evaluation ground truth and the training signal that produces the very lessons fed back into future prompts, which is a fairly strong assumption to be making twice over with the same imperfect instrument. The third is that comparing the with_memory and with_memory_sa conditions in isolation attributes the entire observed difference to the SA module specifically, which, as discussed in section 4.3 below, does not actually hold given how the retrieval depth is configured in the code. The fourth is that a single F1 score, computed at one fixed operating point, is a sufficient summary of whether the model has become better calibrated; the selective-prediction literature, for instance Kamath, Jia and Liang's 2020 paper on selective question answering under domain shift, generally evaluates this exact class of problem using a risk-coverage curve rather than a single point, precisely because one point cannot distinguish a genuinely better signal from a system that has simply moved along the same trade-off curve. The fifth assumption is that the train-test split and the "unanswerable" label set are themselves clean and representative of what the benchmark authors originally intended, which, as section 4.6 shows, does not fully hold either. Finally, the system assumes that the one-word domain tags generated by the model itself are consistent enough across episodes for the domain-based grouping in the promotion mechanism to mean anything at all.

## 4. Problems, weak points and methodological risks

These are presented roughly in order of how much they change the meaning of the headline numbers, the most consequential ones first.

### 4.1 The reported gain is consistent with the model simply hedging more, rather than becoming better at telling the two cases apart

Looking at the raw per-item log for SelfAware, the shift across conditions is quite stark. Going from baseline to RB to RB+SA, the counts of CORRECT, FACTUAL_ERROR, FALSE_IDK and KNOWLEDGE_BOUNDARY_MISSED move as follows: 184, 162, 57 and 97 under baseline; 190, 130, 110 and 70 under RB alone; and 206, 99, 141 and 54 under RB+SA. In other words, FALSE_IDK, which counts a wrongly declined but perfectly answerable question, very nearly triples, and the overall abstention rate rises from about twenty-three percent to about forty-eight percent of the full five hundred test questions. The very same shape appears across all four of the headline datasets, where accuracy on the "should-answer" side of the ledger collapses even as the reported F1 climbs: on SelfAware, answerable accuracy falls from 83.5 to 59.2 percent; on FalseQA, true-premise accuracy falls from 62.2 to 50.0 percent; on KUQ, known-question accuracy falls from 82.7 to 60.4 percent; and on UnknownBench, answer accuracy falls from 94.0 to 58.9 percent, all comparing baseline against RB+SA.

Because the F1 formula as written only scores the abstain side of the decision, this real cost on the answering side is entirely invisible in the single number that gets reported. It is also worth noticing that the two largest claimed gains, KUQ at plus 0.138 and UnknownBench at plus 0.271, are exactly the two datasets where the baseline recall on the abstain class was weakest to begin with, which is exactly where indiscriminately abstaining more often would be expected to pay off most under this particular metric. This was, at first, only a hypothesis formed from reading their logs, but sections 6a and 6b below test it directly on two independent datasets, and both results turn out to be consistent with it: RB+SA cannot be statistically distinguished from a placebo condition carrying the same amount of failure-lesson content retrieved for an entirely unrelated question, with a p-value of 0.428 on SelfAware and 0.719 on FalseQA, the latter being a dataset on which RB+SA cannot even be distinguished from doing nothing at all, with a p-value of 1.000.

### 4.2 The detector, the training signal and the injected content all share the same vocabulary

The lessons and standing guidelines that eventually get injected into the prompt are themselves phrased using more or less the same hedge words that the scoring function is looking for, phrases such as "say I don't know" or "cannot be determined". A model that has been primed with this kind of phrasing could quite easily satisfy the detector simply by echoing its surface form, regardless of whether it has actually reasoned correctly about the specific question in front of it. This offers a plausible mechanism for the pattern described in section 4.1, rather than being a mere coincidence sitting alongside it.

### 4.3 The comparison between RB and RB+SA is not a clean ablation

In both run_selfaware.py and run_falseqa.py, the with_memory condition retrieves the top three ReasoningBank strategies, whereas the with_memory_sa condition retrieves only the top two, the difference being made up with SA content. This means that whatever is being called the "SA uplift", computed as RB+SA minus RB, is also quietly changing the depth of RB retrieval at the same time, so the observed difference cannot be attributed to the SA module on its own as the experiment is currently structured.

### 4.4 The two largest claimed gains, on KUQ and UnknownBench, ship with no code at all

The README lists a file called run_unknownbench.py, but it does not exist anywhere in the folder that was shared, and there is no run_kuq.py either. Only run_selfaware.py and run_falseqa.py are actually present and runnable. This means there is simply no way to inspect how the other two headline numbers, which happen to be the two largest reported gains, were actually produced.

### 4.5 Of the twelve reported model and dataset combinations, only two are reproducible from what was shared

The checkpoints folder and the openrouter_experiments folder together show that the same four datasets were also run against Gemini and Qwen3, which, together with Qwen2.5-7B, accounts for the "eleven out of twelve" figure mentioned in the covering email, three models across four datasets making twelve combinations in total. However, there is no source code anywhere for the Gemini or Qwen3 runs, no OpenRouter client and no run scripts, only the raw checkpoint files themselves. While going through these, the likely candidate for the one combination out of twelve that did not improve was also found: Gemini on FalseQA is actually worse under RB+SA than under baseline, with F1 falling from 0.443 to 0.391.

### 4.6 Benchmark runs that do not fit the reported story are simply absent from the README

There are several result files sitting in the folder that are never mentioned anywhere in the README or the brief. One of these, abstentionbench_summary.json, reports an F1 of zero across all three conditions, because the model abstained precisely zero times out of three hundred, in every single condition. AbstentionBench, published by Meta FAIR and available at arXiv 2506.09038, is a genuine and fairly recent twenty-dataset benchmark built specifically to stress-test abstention behaviour, and KUQ, which is one of this project's own headline datasets, is in fact one of its constituent sub-benchmarks. This arguably makes it the single most relevant existing benchmark for the exact claim being made here, and the result on it is silently zero. A second file, hle_summary.json, shows an accuracy that is bit for bit identical, at fifty-three out of two hundred correct, across baseline, RB and RB+SA, meaning zero uplift of any kind. A third, awarebench_summary.json, shows RB+SA performing worse than baseline by 7.5 points, only partially recovering the drop that RB alone had already caused. And a fourth, selfaware_v3_summary.json, appears to be a rerun of the very same nominal benchmark that produces a different F1, 0.524 rather than the reported 0.506, and also includes an additional condition called with_memory_sa_selfmodel that performs worse than everything else in the file. None of this necessarily makes the reported numbers false, but it does mean that the "eleven out of twelve" framing undercounts how many combinations were actually attempted, and the ones that do not fit the story are simply not discussed anywhere.

### 4.7 Some of the "unanswerable" training labels are corrupted rather than genuinely unanswerable

The unanswerable questions sourced from the original SelfAware dataset, as published by Yin and colleagues at ACL 2023, are for the most part genuine, being subjective or philosophical questions with no settled answer, of the kind "do truly good people exist". Mixed in among these, however, are a number of corrupted GSM8K-style word problems, for instance one that begins by talking about bears and then, midway through, refers to putting "the dogs" onto shelves, and another that asks how many points were earned "for every unknown pounds" that were recycled. These read like an automated attempt to manufacture synthetic unanswerable questions by substituting a quantity with the word "unknown", carried out with visible substitution errors along the way. Such questions are trivially unanswerable simply because the sentence itself does not parse, rather than because they test any genuine epistemic boundary, and this muddies what the unanswerable-accuracy figures are actually measuring.

### 4.8 The setup instructions do not reproduce as written

The README instructs the reader to start "a flask server", while the comment inside llm_client.py describes the very same client as talking to a "llama-server"; the file simple_server.py, which the README refers to directly, does not exist anywhere in the shared folder, and flask itself is not even installed in the environment. The main function in run_selfaware.py also imports from a module called download_selfaware, which likewise does not exist. None of this is fatal, and a way around it is described in section 6 below, but as shared, the project does not actually run starting from a clean checkout.

### 4.9 A few smaller findings

The check used to decide whether an attempted answer is factually correct, which in turn decides whether an episode is labelled CORRECT or FACTUAL_ERROR, is a fairly loose substring containment, checking simply whether the gold answer appears anywhere within the model's response. This does not affect the headline F1, which only scores the abstain decision, but it does affect which lessons get extracted and eventually promoted into guidelines. Separately, the confidence field on every episode is hardcoded to 0.5 in the extraction code, with a comment noting explicitly that the LLM's own self-reported confidence is not to be trusted, and yet the SelfModel class goes on to compute a "calibration gap" from this same constant, which is numerically meaningless wherever it is actually used; tellingly, the one place this machinery is wired in, the with_memory_sa_selfmodel condition in the selfaware_v3 file, performs worse than every other condition in that file. The memory module also declares nine possible failure types, of which only four are ever actually produced by the evaluation functions in use, leaving the other five as unexercised code paths. On a more reassuring note, the SelfAware training and test sets were checked and found to have no overlapping questions at all between 2021 training items and 1348 test items, which is worth stating since not everything examined during this audit turned out to be a problem; the corresponding check for FalseQA found exactly one duplicate question out of 2838 training and 1892 test items, which is negligible in practice but not literally zero.

## 5. Proposed robustness check

The central open question, following from sections 4.1 and 4.2 above, is whether the RB+SA gain is actually driven by the specific relevance of the retrieved failure lessons to the question being asked, which is the system's own claimed mechanism, or whether any content shaped like a failure lesson would produce much the same shift, simply because the model is being primed to hedge more in general. The check proposed here is to add a fourth condition, called with_memory_sa_scrambled, that is identical to with_memory_sa in every respect, using the same RB retrieval depth of two and the same standing guidelines, except that the SA episodic lesson retrieval is queried using a different, unrelated test question rather than the real one, the pairing being fixed through a derangement of the test set so that no question ever ends up retrieving its own lessons. This is a deliberately minimal, single-variable change: the total amount of injected SA content, its surface phrasing and everything else about the pipeline stay fixed, and the only thing that moves is whether the retrieved content is actually relevant to the question at hand.

This was a conscious choice to keep the scope small, in line with the brief's own guidance that a focused check beats a broad one. A fuller design was considered and then set aside, including a naive "just hedge more" instruction-only control, a version matched for overall abstention rate, and a full risk-coverage curve; these remain reasonable follow-ups and are noted again in sections seven and eight, but the single scrambled-retrieval condition, together with a proper paired significance test, was judged to answer the sharpest open question with the least additional surface area. If with_memory_sa and with_memory_sa_scrambled turn out to be statistically indistinguishable under a paired test on the same held-out items, that would be strong evidence that the gain is not coming from the relevance of the retrieved lessons but from their mere presence. If with_memory_sa is significantly better, that would be genuine evidence for the mechanism the system actually claims to rely on.

## 6. What was implemented

The changes described here sit in sa-module-v2/src/run_selfaware.py and sa-module-v2/src/run_falseqa.py, and can be seen in commits 0674b21 and 161a800 in the repository history. A function called build_derangement was added, producing a fixed and reproducible permutation with no fixed points, used to pair each test question with an unrelated one for the scrambled condition. A new branch was added inside the test loop of each script for the with_memory_sa_scrambled condition, structurally identical to with_memory_sa apart from the query used for SA retrieval. A paired, continuity-corrected McNemar's test was also added, since every condition in this experiment runs on the same ordered test set, which makes it a paired design that a plain comparison of overlapping confidence intervals would not properly respect. The results-saving step now reports the abstention rate for each condition and runs McNemar's test between with_memory_sa and with_memory_sa_scrambled, which is the central comparison, as well as between with_memory_sa and baseline and between with_memory and baseline for context. Command-line arguments for the training and test limits were also added to run_selfaware.py, defaulting to the original 300 and 500 used in the paper, so the check can be run either at full scale or at a faster, reduced scale.

It should be mentioned that the original memory-bank checkpoint used to produce the numbers reported in the README was not included anywhere in the shared materials, only the final summaries and one raw per-item log having survived. Because of this, the run described here rebuilds the RB and SA memory from the training set entirely from scratch, which has the useful side effect of also serving as an independent reproduction check of the reported baseline, RB and RB+SA numbers on SelfAware, before the new scrambled condition is added on top of that. On a practical note, since the shared setup instructions do not run as written, as discussed in section 4.8, an actual OpenAI-compatible server was instead stood up directly, using the command "vllm serve Qwen/Qwen2.5-7B-Instruct" with the served model name set to match what llm_client.py expects by default, which meant the existing client code could be used entirely unmodified.

### 6a. Results on SelfAware

A full-scale reproduction was run with training limited to 300 and testing limited to 500, matching the paper's own setup exactly, with the memory rebuilt from scratch since no checkpoint was available. The results were as follows: baseline produced an F1 of 0.427 with precision 0.492 and recall 0.377; RB alone produced an F1 of 0.502 with precision 0.455 and recall 0.558; RB+SA produced an F1 of 0.524 with precision 0.442 and recall 0.643; and RB+SA with scrambled SA retrieval produced an F1 of 0.495 with precision 0.422 and recall 0.597. Answerable accuracy fell steadily across these same conditions, from 82.7 percent under baseline to 70.2 under RB, 63.9 under RB+SA and 63.6 under the scrambled version, while unanswerable accuracy rose from 37.7 to 55.8, 64.3 and 59.7 percent respectively, and the overall abstention rate climbed from 23.6 to 44.8 percent between baseline and RB+SA.

Running a paired McNemar's test on these same five hundred items, comparing RB+SA against the scrambled version gives b equal to 43, c equal to 35, a chi-square statistic of 0.628 and a p-value of 0.428, which is not significant. Comparing RB+SA against baseline gives b equal to 54, c equal to 78, a chi-square statistic of 4.008 and a p-value of 0.045, which is significant. Comparing RB alone against baseline gives a p-value of 0.203, which is again not significant.

Two things follow from this. First, the overall pattern of baseline being worse than RB, which is in turn worse than RB+SA, does replicate independently from a fresh training run, the numbers 0.427, 0.502 and 0.524 sitting close to the reported 0.425, 0.483 and 0.506, so the direction of the reported effect appears to be genuine rather than a one-off or fabricated result. Second, and more importantly, RB+SA beats the scrambled placebo by only 0.029 F1, and this difference cannot be distinguished from noise at this sample size, whereas RB+SA against doing nothing at all is significant. What this suggests is that the system's own claim, namely that retrieving relevant past failures is what helps, is not what the data actually supports; what the data does support is that adding content shaped like a failure lesson shifts behaviour, largely regardless of whether that content has anything to do with the question being asked. One caveat is worth stating plainly here: the scrambling only touched the per-episode retrieval and left the twelve standing guidelines that were promoted during training untouched, since these are identical in both the real and the scrambled condition. This experiment therefore does not, on its own, say anything about whether the guidelines themselves are doing real work, and scrambling or removing those as well would be a natural next step.

### 6b. Results on FalseQA

The same ablation was applied to run_falseqa.py at full scale, with training and testing both limited to 300, again matching the paper's own setup. Here, baseline produced an F1 of 0.673 with precision 0.621 and recall 0.735; RB alone produced an F1 of 0.676 with precision 0.534 and recall 0.919; RB+SA produced an F1 of 0.729 with precision 0.584 and recall 0.971; and the scrambled version produced an F1 of 0.726 with precision 0.575 and recall 0.985. False-premise accuracy rose across these conditions from 73.5 to 91.9, 97.1 and 98.5 percent, while true-premise accuracy fell from 62.8 to 33.5, 42.7 and 39.6 percent respectively.

The paired McNemar's test here tells an even sharper story than SelfAware did. Comparing RB+SA against the scrambled version gives b equal to 17, c equal to 14, a chi-square of 0.129 and a p-value of 0.719, again not significant. Comparing RB+SA against baseline gives b equal to 42, c equal to 43, a chi-square of essentially zero and a p-value of 1.000, meaning RB+SA cannot be distinguished from doing nothing at all on this dataset. The only comparison that clears significance here is RB alone against baseline, with a p-value of 0.017.

Once again, the reproduction lands close to the reported figures, 0.673, 0.676 and 0.729 against their reported 0.680, 0.674 and 0.736, so the effect direction replicates a second time. But the ablation result is, if anything, stronger evidence against the relevance claim than SelfAware's was: the gap between RB+SA and the scrambled placebo is a mere 0.003 F1, and on this dataset RB+SA is not even distinguishable from baseline. It is also worth noting, purely as an honest observation, that the direction of the small real-versus-scrambled gap actually flips between the two datasets, real SA showing a slightly higher abstention rate than scrambled on SelfAware but a slightly lower refusal rate than scrambled on FalseQA, which is itself consistent with these being noise-level differences rather than any systematic effect of retrieval relevance. Taken together across both datasets for which runnable code exists, the same conclusion holds throughout: the claim that relevant retrieval of past failures is what drives the reported gain is not supported once it is actually tested. What the data is consistent with instead is a real, if modest and somewhat dataset-dependent, effect of adding memory-derived content to the prompt in general, on top of which the SA-specific mechanism, which is the part of the system actually being presented as the contribution, does not clear the bar of a same-population placebo control.

## 7. How to reproduce this work

The following steps were used and should reproduce the results above on a machine with a spare GPU. From within the sa-module-v2 directory, create the data, memory_bank and results folders, and copy the selfaware_train.json and selfaware_test.json files into the data folder. A model server matching what llm_client.py expects can then be started using vllm, specifying the served model name explicitly so that it matches the client's default, along with a bfloat16 data type, a conservative GPU memory utilisation setting and a modest maximum model length. With the server running, the full-scale run matching the paper's reported setup can be started with a training limit of 300 and a test limit of 500, which takes roughly an hour to an hour and a half on a single H200, or a faster check can be run with smaller limits such as 100 and 150. Passing a flag to skip the scrambled condition will reproduce only the original three conditions. The same ablation can be run on FalseQA by copying its train and test files into the data folder in the same way and running run_falseqa.py directly, its limits being set inside the script itself. The outputs land in results/selfaware_summary.json and results/falseqa_summary.json, each containing all four conditions along with the full breakdown and the McNemar test results, alongside the raw per-item logs in results/selfaware_results.json and checkpoints/falseqa.

## 8. A complementary research direction

The results in sections 6a and 6b point toward a fairly specific gap in the current design. The system's only signal for whether a model has expressed uncertainty is a keyword match against the surface text of its response, and this same signal is what generates the training lessons that are then fed back into future prompts. What sections 6a and 6b together show is that this loop can be fooled, in the sense that behaviour which looks identical to genuine calibration under this particular detector was reproducible, on two separate datasets, using lessons retrieved for entirely the wrong question. Both the current RB+SA mechanism and the clustering and routing idea that Nhat is exploring in parallel operate entirely within this same text space, whether that is the surface phrasing of a response or the lexical and embedding similarity between questions, and neither one looks at what is actually happening inside the model when it produces an answer, which means neither one is in a position to independently check whether the lexical detector is being fooled.

A complementary direction worth pursuing, then, would be to bring in a model-internal uncertainty signal that does not depend on what the model chooses to say out loud, and to use it as an independent channel against which the lexical detector can be validated, or eventually replaced. Semantic entropy, introduced by Farquhar and colleagues in a 2024 Nature paper, works by sampling several generations for the same question, clustering them by semantic equivalence rather than surface wording, and computing entropy over these clusters; a question the model is genuinely uncertain about tends to produce semantically scattered answers even when each individual answer, read in isolation, sounds perfectly confident. A follow-up method known as Semantic Entropy Probes, due to Kossen and colleagues, shows that this same quantity can be approximated fairly cheaply from the hidden states of a single generation, which avoids the cost of sampling multiple completions for every question. Concretely, this would allow one to check whether the existing lexical detector agrees with semantic entropy on the same set of responses, and if the two diverge often, that would be fairly direct evidence that the keyword detector is picking up on surface form rather than genuine uncertainty, which is exactly the mechanism the null results in sections 6a and 6b are consistent with. It would also allow the hardcoded confidence value of 0.5 to be replaced with a real, non-circular estimate, which would in turn make the calibration-gap computation inside SelfModel, currently meaningless as described in section 4.9, actually mean something. Eventually, this could support moving away from a single F1 operating point altogether, in favour of a proper risk-coverage curve of the kind used in the selective-prediction literature, which is really the only way to cleanly separate a genuinely better underlying signal from a system that has simply moved along the same trade-off curve, an ambiguity this entire audit has, in one way or another, kept circling back to.

## 9. Questions I would want to clarify

The first thing I would want to ask is whether an internal ablation has ever been run comparing retrieval that is relevant against retrieval that is not, since the scrambled-retrieval placebo described above could not statistically distinguish RB+SA from a version fed lessons retrieved for the wrong question, on either of the two datasets that could actually be run, and on FalseQA specifically, RB+SA could not even be distinguished from baseline. Taken together, this is consistent with most of the measured effect coming from the standing guidelines or from a general increase in hedging, rather than from the relevance of retrieval as such, which would be a somewhat different story from the one told in the README, and I would be curious whether this matches what has been seen internally, or whether KUQ and UnknownBench, where the largest gains are reported, tell a different story altogether. Second, I would want to know where the code for KUQ and UnknownBench actually is, and likewise for the Gemini and Qwen3 runs, and whether their absence from what was shared was simply an oversight or was intentional given the scope of this exercise. Third, I would want to know whether abstentionbench was left out of the README because it is a known-broken evaluation, or simply because it had not yet been written up, and the same question applies to hle, awarebench and selfaware_v3. Finally, I would want to know whether improving F1 on the abstain class is genuinely the objective being optimised for here, or whether a metric that penalises over-abstention symmetrically, such as a macro F1 across both classes or a proper risk-coverage curve, would better reflect what "more trustworthy" is actually meant to capture for this project.
